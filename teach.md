# teach.md — 面接 / 発表での説明用ガイド

このポートフォリオサイトを「人に説明する」ための資料。
そのまま読み上げてもいいし、面接で深掘りされたとき用のネタ帳としても使える。
上から順に「30秒 → 3分 → 深掘り」と粒度が細かくなる構成。

---

## 1. 30秒で言うなら（エレベーターピッチ）

> 「React 19 + Vite で作った個人ポートフォリオです。特徴は、**PC とスマホでまったく違う UX を、同じコンポーネント群から出し分けている**こと。PC は縦スクロール1ページ、スマホは Hero をランディングにして、タップでセクションを切り替える SPA 風のページ遷移にしています。ルーティングは外部ライブラリを使わず、URL のハッシュと自作フックだけで実装しました。」

ポイントは3つだけ覚えておけばいい:
1. **React 19 + Vite + CSS Modules**（モダンで軽量な構成）
2. **PC / スマホで UX を分ける設計**（ただ縮めるだけのレスポンシブではない）
3. **ライブラリに頼らず自作フックで実装**（仕組みを理解している証明）

---

## 2. 3分版（技術の全体像）

### 使った技術と、なぜそれを選んだか

| 技術 | 選んだ理由（面接で聞かれたら） |
|------|------|
| **React 19** | 最新の安定版。コンポーネント分割で UI を部品化したかった |
| **Vite 8** | ビルド・開発サーバーが高速。設定がほぼ不要で個人開発に向く |
| **CSS Modules** | クラス名の衝突を自動で防げる。コンポーネント単位でスタイルを閉じ込められる |
| **gh-pages** | サーバー不要・無料で公開できる。`npm run deploy` 一発でデプロイ |
| **ライブラリ不使用のルーティング** | React Router を入れるほどの規模ではなく、ハッシュで十分と判断 |

> 補足: react-router や styled-components を「あえて入れなかった」。
> 「規模に対して過剰だから自作した」と言えると、技術選定の判断力をアピールできる。

### アーキテクチャの核：PC / スマホの出し分け

入口の `src/App.jsx` で `useIsMobile()` を呼び、true / false で表示を分岐:

- **PC** → 全セクションを縦に並べた1ページ（普通のポートフォリオ）
- **スマホ** → `<MobileRouter />` に丸ごと委譲し、ページ遷移型 UI に切り替え

```jsx
const isMobile = useIsMobile()
// ...
{isMobile ? <MobileRouter /> : <>全セクション縦並び</>}
```

## 3. 深掘り用ネタ（聞かれたら強い3つ）

### ネタ① 自作フック `useMediaQuery`（PC/スマホ判定）

ブラウザの `window.matchMedia` を React の state に橋渡しするフック。
画面幅が変わった瞬間（PC↔スマホの境界 768px をまたぐ）に自動で再レンダリングされる。

```js
export const useIsMobile = () => useMediaQuery('(max-width: 768px)')
```

**語れるポイント:**
- 初期値を `useState(() => …)` の関数形式で渡し、初回レンダリングでも正しい判定を返す
- `useEffect` で `change` イベントを購読し、**クリーンアップ（removeEventListener）も忘れずに**書いた → メモリリーク対策を理解している
- `typeof window !== 'undefined'` ガードで、サーバー環境でも落ちないようにしている

### ネタ② 自作のハッシュルーティング `useHashRoute`（スマホのページ遷移）

スマホでは Hero がトップ画面。リンクをタップすると URL が `#about` のように変わり、
それを検知して対応するセクションだけを表示する。React Router は使っていない。

```js
const onChange = () => setHash(window.location.hash.replace('#', ''))
window.addEventListener('hashchange', onChange)
```

`MobileRouter.jsx` 側で、ハッシュ文字列をコンポーネントに対応づけるマップを持つ:

```js
const routes = { about: About, skills: Skills, projects: Projects, blog: Blog, contact: Contact }
```

**語れるポイント:**
- ハッシュ → コンポーネントの**辞書（オブジェクト）引き**で分岐。if 文を増やさず、追加に強い設計
- 不正なハッシュ（`#xxxx`）は Hero にフォールバックさせ、壊れない
- ルート変更時に `window.scrollTo` で**ページ先頭へスクロール**を再現 → ネイティブアプリっぽい遷移体験
- 「← Back」は `history.pushState` でハッシュを消し、ホーム（Hero）へ戻す

### ネタ③ コンポーネントの PC / スマホ分割パターン

各セクションは `index.jsx` が振り分け役で、中身を `.desktop.jsx` と `.mobile.jsx` に物理分割している。

```
src/components/Hero/
├── index.jsx            ← useIsMobile で desktop/mobile を出し分け
├── Hero.desktop.jsx     ← PC 専用
└── Hero.mobile.jsx      ← スマホ専用
```

**語れるポイント:**
- 「CSS のメディアクエリで全部やる」のではなく、**JSX 自体を分けた**。
  PC とスマホで構造（DOM）が大きく違うため、1ファイルに詰め込むと条件分岐だらけになると判断した
- おかげで PC 担当 / スマホ担当を **別 git ブランチで並行開発**でき、コンフリクトが起きにくい
  （`feature/desktop` は `*.desktop.*` のみ、`feature/mobile` は `*.mobile.*` のみ編集）

## 4. 想定 Q&A（突っ込まれたとき用）

**Q. なぜ React Router を使わなかった？**
A. ページ数が少なく、スマホのセクション切替だけが対象だったので、URL ハッシュと自作フックで十分でした。依存を増やさず、ルーティングの仕組み自体を理解する練習にもなりました。

**Q. レスポンシブとどう違う？**
A. 一般的なレスポンシブは同じ DOM を CSS で見た目だけ変えます。今回は **UX の設計自体を PC とスマホで変えて**いて、スマホは縦スクロールではなくページ遷移型にしています。そのため JSX レベルで分割しました。

**Q. パフォーマンスや今後の課題は？**
A. 現状の課題は `teach.md` 末尾の TODO（i18n の実装、画像差し替え、Blog の中身）です。多言語切替は今は UI だけで、`react-i18next` などの導入を検討しています。

**Q. 一番苦労した / 工夫した点は？**
A. スマホのページ遷移を「ライブラリなしで、戻るボタンやスクロール位置までネイティブアプリっぽく」作り込んだ部分です（上記ネタ②）。

## 5. デモの流れ（発表で実演するなら）

1. PC ブラウザで開く → 縦スクロールで全セクションを見せる
2. ブラウザ幅を縮める（または DevTools のスマホ表示）→ **その場で UI がページ遷移型に切り替わる**のを見せる ← ここが見せ場
3. Hero のリンクをタップ → URL が `#projects` に変わる → セクションが切り替わる
4. 「← Back」でホームに戻る

> 2 の「幅を変えた瞬間に切り替わる」は `useMediaQuery` のイベント購読が効いている証拠。実演すると分かりやすい。
